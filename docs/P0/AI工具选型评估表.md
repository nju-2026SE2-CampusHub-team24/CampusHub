# AI工具选型评估表

## 一、文档目的

为规范 CampusHub（校园互助服务平台）项目中的 AI 工具使用方式，提升需求分析、系统设计、编码实现、代码审查、测试与文档整理等工作的效率，同时保证项目过程合规、结果可审查、责任可追溯，团队依据课程要求、项目特点及成员实际情况，对候选 AI 工具进行了选型评估，并形成如下结论。

本评估表用于说明团队在项目开发过程中“选择哪些 AI 工具、用于哪些环节、为什么选择、存在哪些风险以及如何控制”，作为后续 AI 协作实践与阶段性反思的重要依据。

---

## 二、选型依据

结合课程总览、团队章程及《AI 协作契约》，本团队在 AI 工具选型时主要依据以下原则：

1. **工具性原则**  
   AI 是辅助工具，不替代团队成员的独立思考、工程判断与最终责任承担。

2. **可审查原则**  
   所有 AI 输出均须经过人工检查、修改与确认后，方可用于项目文档、代码或测试过程。

3. **适配性原则**  
   选用工具应与课程项目的任务类型、团队成员当前能力基础及协作方式相匹配，避免盲目追求工具数量与复杂度。

4. **统一性原则**  
   优先选择团队成员普遍能够使用、便于统一管理与记录的工具，减少多工具并行带来的沟通与留痕成本。

5. **风险可控原则**  
   对涉及核心业务逻辑、数据库设计、安全相关实现等关键内容，应严格限制 AI 的直接使用范围，确保最终方案由人工主导并验证。

---

## 三、评估维度

本次 AI 工具评估主要从以下维度展开：

- **场景适配度**：工具是否适用于需求分析、设计、开发、代码审查、测试、文档整理等课程项目常见场景
- **输出质量**：工具生成内容是否清晰、可理解、可修改、可复用
- **协作便利性**：是否便于团队统一使用、共享经验与保留记录
- **学习成本**：团队成员是否能够在较短时间内完成上手
- **风险可控性**：输出是否便于人工审查、纠错与验证
- **课程契合度**：是否有利于满足课程对 AI 使用规范、过程记录与反思日志的要求

| ChatGPT / Copilot 辅助测试 | AI 测试辅助方式 | 测试思路补充、边界情况提示、基础测试代码补全 | 可帮助团队补充测试覆盖思路，降低测试阶段入门门槛 | AI 无法保证测试结果真实性，测试执行与结论必须人工完成 | 采用 |

## 五、最终选型方案

经综合评估，团队确定本项目采用以下 AI 工具链：

### 5.1 主力工具
1. **ChatGPT**  
   用于需求分析、功能拆解、文档初稿生成、技术问题答疑、代码解释及测试思路补充等工作。

2. **GitHub Copilot**  
   用于编码阶段的代码补全、样板代码生成、常见接口与测试代码辅助编写。

3. **CodeRabbit**  
   用于团队协作开发中的 Pull Request 审查辅助，帮助检查代码修改中的明显问题、潜在风险和可改进点。

### 5.2 补充工具
1. **Google Gemini**  
   用于对部分分析结果进行交叉验证，并在文档表达、内容组织等方面提供补充参考。

2. **ChatGPT 辅助代码审查**  
   在人工 Code Review 基础上，用于帮助团队成员理解复杂代码、发现明显逻辑问题与辅助排查 Bug。

### 5.3 暂不采用工具
- Cursor
- 通义灵码
- CodeGeeX 等其他 AI 编程助手

暂不采用上述工具的主要原因在于：当前项目阶段更需要保证工具链统一、使用稳定、团队协作成本可控，而非追求工具种类的丰富性。

## 六、选型理由说明

### 6.1 选择 ChatGPT 的原因
ChatGPT 适用范围广，能够覆盖需求分析、设计、开发、测试与文档整理等多个阶段。对于当前团队而言，其最大的价值在于能够帮助成员快速完成问题拆解、形成初稿、理解陌生概念，并为项目推进提供较强的过程支持。尤其是在团队成员项目经验相对有限的情况下，ChatGPT 具有较高的通用性与实用性。

### 6.2 选择 Gemini 作为补充工具的原因
Gemini 已被纳入团队章程中的 AI 对话工具选项。将其作为补充工具使用，有助于在关键分析结论、文档表述或问题理解上进行交叉验证，降低单一工具回答偏差带来的风险。但考虑到团队协作一致性要求，Gemini 不作为主要工作工具。

### 6.3 选择 GitHub Copilot 的原因
GitHub Copilot 能够直接嵌入开发环境，在实际编码中提供实时补全与建议，较适合本项目中的常见 CRUD 逻辑、接口编写、基础测试代码生成等场景。其优势在于能够在不改变团队主要开发流程的前提下，提升编码效率并减少重复劳动。

### 6.4 选择 CodeRabbit 的原因
CodeRabbit 主要用于辅助代码审查，适合团队在多人协作开发过程中对 Pull Request 进行初步检查。它能够对代码变更给出结构化的审查建议，帮助团队更快发现明显问题、潜在缺陷以及代码风格或可维护性方面的不足。对于课程项目而言，CodeRabbit 的价值主要体现在“提高审查效率”和“补充人工审查视角”两个方面。

### 6.5 暂不选择其他 AI 编程工具的原因
尽管其他 AI 编程助手在项目级理解、跨文件问答等方面具有一定优势，但团队当前更重视“统一使用、便于记录、降低学习与切换成本”。综合考虑后，当前阶段采用 Copilot 负责编码辅助、CodeRabbit 负责审查辅助，更符合项目推进效率与团队协作稳定性的要求。

## 七、风险分析与控制措施

| 风险类型 | 风险表现 | 控制措施 |
|---|---|---|
| AI 输出不准确 | 需求理解偏差、代码逻辑错误、测试场景遗漏 | 所有 AI 输出必须经人工检查；关键内容由相关负责人二次确认 |
| 过度依赖 AI | 成员照搬结果但缺乏理解，影响个人能力与项目质量 | 要求成员能够解释自己负责部分的设计与实现；核心内容不得机械复制 |
| 输出与项目上下文不匹配 | 代码、文档或测试方案脱离 CampusHub 实际需求 | 在提问时明确项目背景与当前阶段；对输出进行本地化修改 |
| 工具过多导致协作混乱 | 输出风格不统一、日志难管理、经验难共享 | 控制工具数量，以 ChatGPT、Copilot 和 CodeRabbit 为主，Gemini 为辅 |
| AI 内容未经验证直接使用 | 文档、代码、测试结果失真，影响项目诚信与质量 | 严格执行人工审查、Code Review、测试验证和日志留存制度 |
| 过度依赖自动化审查结论 | 将 CodeRabbit 的建议视为最终结论，忽略人工判断 | 明确 CodeRabbit 仅作辅助审查，最终是否修改由开发者和审查者共同判断 |

## 八、使用规范

为确保 AI 工具使用过程符合团队章程与《AI 协作契约》，团队在本项目中作如下规定：

1. AI 可用于需求分析、文档初稿、思路整理、样板代码生成、测试思路补充、代码审查辅助与问题答疑。
2. 核心业务逻辑、数据库设计、安全相关实现、正式提交材料等内容必须经过更严格的人工作审查。
3. CodeRabbit 仅用于辅助 Pull Request 审查，不替代人工 Code Review，也不直接决定代码是否合并。
4. 严禁伪造调研材料、伪造测试结果或直接提交未经检查的 AI 生成内容。
5. 所有重要 Prompt、AI 输出与审查建议需保留记录，并用于后续阶段性《AI 协作反思日志》的撰写。
6. 团队成员应对最终提交的文档、代码与测试结果负责，AI 不构成责任主体。

---

## 九、结论

综合项目特点、团队能力基础、课程要求与协作便利性等因素，团队最终确定采用“**ChatGPT 为主、Google Gemini 为辅、GitHub Copilot 支持编码、CodeRabbit 辅助代码审查**”的 AI 工具链方案。

该方案能够较好覆盖需求分析、系统设计、编码实现、代码审查、测试辅助与文档整理等项目关键环节，既有助于提高团队开发效率，也有利于保证 AI 使用过程的规范性、可审查性与可追溯性，符合课程关于 AI 协作实践的总体要求。
