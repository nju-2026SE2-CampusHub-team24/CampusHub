## 项目名称：CampusHub（校园互助服务平台）
**版本号：** V1.0  
**编制日期：** 2026年4月  


---

# 1. 引言

## 1.1 目的
本文档用于定义 CampusHub 校园互助服务平台的功能需求与非功能需求，为后续系统设计、数据库设计、接口设计、编码实现、测试与验收提供统一依据。  


## 1.2 范围
CampusHub 是一个面向本校大学生的校园互助服务平台，支持学生发布互助需求、接单履约、沟通协作、确认完成、互评打分，并在发生争议时申请平台介入处理。  

系统重点覆盖以下校园场景：

- 跑腿/代取
- 学习辅导/答疑
- 二手交易
- 活动组队/找搭子
- 资源共享/资料获取
- 紧急求助

系统支持以下三类角色：
- 游客
- 已认证学生
- 平台管理员

其中，已认证学生可以在不同场景下分别扮演需求发布者或任务接受者。

## 1.3 定义、术语与缩略语
- **SRS**：软件需求规格说明书（Software Requirements Specification）
- **游客**：未完成校园认证的普通访问用户，可进行基础浏览与注册登录
- **已认证学生**：完成校园身份认证的用户，可执行平台核心互助功能
- **平台管理员**：负责审核、治理、纠纷处理和用户状态管理的后台角色
- **需求方**：在某次具体任务中发布互助需求的已认证学生
- **服务方**：在某次具体任务中接受并完成任务的已认证学生
- **校园认证**：通过学号、校园邮箱、学生证等方式验证用户为本校学生
- **信用分**：系统根据履约、评价、违规情况形成的信用指标
- **历史评价**：互助完成后用户之间留下的评分与文字反馈
- **仲裁**：当用户对交易或互助结果存在争议时，由平台管理员介入处理

# 2. 总体描述

## 2.1 产品视角
CampusHub 是一个独立运行的校园 Web 应用系统，采用前后端分离架构。根据团队 README，系统当前技术栈为：

- 前端：Vue 3 + Element Plus
- 后端：Spring Boot（Java）
- 数据库：MySQL
- CI：GitHub Actions

系统将主要由以下部分组成：
- 学生端前台
- 管理员后台
- 后端业务服务
- 数据库存储



## 2.2 用户类别与特征

### 2.2.1 游客
- 可浏览平台公开内容
- 可注册与登录
- 可发起认证流程
- 不可执行核心互助行为

### 2.2.2 已认证学生
- 是平台的核心使用者
- 既可能发布需求，也可能接受需求
- 关注流程是否快捷、对方是否可靠、状态是否透明
- 希望平台提供接单、沟通、评价、信用等完整闭环能力

### 2.2.3 平台管理员
- 审核校园认证申请
- 处理交易纠纷与申诉
- 管理违规内容
- 管理用户状态（如封禁、信用处理）

## 2.3 产品功能概述
系统主要包括以下模块：

1. 注册与登录模块
2. 校园认证模块
3. 需求发布模块
4. 紧急求助模块
5. 大厅浏览与分类检索模块
6. 接单与履约模块
7. 站内沟通与议价模块
8. 任务完成确认与结算模块
9. 双向评价与信用模块
10. 仲裁与申诉模块
11. 管理后台模块

## 2.4 设计约束
1. 核心互助能力仅对已认证学生开放。
2. 平台应优先实现课程项目核心闭环，不追求商业化复杂度。
3. 平台应围绕用例图中的主流程组织功能，避免需求发散。
4. 平台设计应便于按模块分工开发和后续扩展。
5. 平台必须对用户隐私、违规内容和信用治理给予基本支持。

## 2.5 假设
1. 用户愿意在校内互助场景中使用该平台。
2. 用户能接受一定程度的校园认证，但平台需控制敏感信息暴露。
3. 平台一期重点是“发布—接单—沟通—完成—评价—申诉”的核心闭环。
4. 游客与已认证学生之间存在明确权限边界。

# 4. 非功能需求

## 4.1 性能需求

### NFR-P-01 核心操作响应快
系统应保证注册登录、浏览大厅、发布任务、接单、查看状态等核心操作具有较快响应。  
**优先级：** Must  
**来源：** `[AI生成+人工修改]`  
**修改说明：** AI 初稿给出了较强工程指标，本版保留“快速响应”目标但不硬绑定过重数值。
### NFR-P-02 匹配效率高
平台应尽可能缩短任务从发布到获得响应的时间。  
**优先级：** Must  
**来源：** `[人工补充]`

## 4.2 安全需求

### NFR-S-01 敏感信息保护
系统应保护密码、学号、真实姓名、取件码等敏感信息，避免无关泄露。  
**优先级：** Must  
**来源：** `[AI生成+人工修改]`

### NFR-S-02 访问权限控制
系统应根据角色区分游客、已认证学生和平台管理员的操作权限。  
**优先级：** Must  
**来源：** `[人工补充]`  
**补充依据：** 用例图清楚划分了三类角色及权限边界。

### NFR-S-03 内容合规与社区治理
系统应支持违禁内容拦截、内容审核和违规处理，维护良好平台生态。  
**优先级：** Must  
**来源：** `[AI生成+人工修改]`

## 4.3 可用性需求

### NFR-U-01 流程快捷
用户应能用较少步骤完成注册、认证、发布任务、接单、确认完成等操作。  
**优先级：** Must  
**来源：** `[人工补充]`

### NFR-U-02 界面清晰
页面应便于用户理解角色权限和任务流程。  
**优先级：** Should  
**来源：** `[人工补充]`

### NFR-U-03 反馈及时
当状态变更、任务完成、收到评价或仲裁有结果时，系统应及时反馈。  
**优先级：** Must  
**来源：** `[人工补充]`

## 4.4 可维护性需求

### NFR-M-01 模块化设计
系统应按认证、任务、订单、评价、仲裁、管理等模块清晰拆分。  
**优先级：** Should  
**来源：** `[AI生成+人工修改]`

### NFR-M-02 文档与接口规范
系统应提供必要的开发文档、接口说明和部署说明。  
**优先级：** Should  
**来源：** `[AI生成+人工修改]`
