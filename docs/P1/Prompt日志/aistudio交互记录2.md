1. 引言 (Introduction)
1.1 编写目的 (Purpose)

本文档旨在详细描述“校园互助服务平台”的软件需求。本文档将作为平台设计、开发、测试以及验收的基础依据，确保开发团队、项目管理者与利益相关者（如高校学生会、平台运营方）对软件功能及非功能需求达成一致理解。

1.2 文档范围 (Scope)

本系统名为**“校园互助服务平台”**。
该平台主要面向在校大学生，旨在打破校园内的信息孤岛，建立一个安全、高效、便捷的校园供需匹配社区。平台支持学生作为“需求方”发布任务，或作为“服务方”接单。涵盖的核心业务场景包括：快递代取、学习辅导、二手交易和组队匹配等。系统包含移动端（微信小程序/App）供学生使用，以及Web端后台供管理员进行内容审核与用户管理。

1.3 定义、首字母缩写词和缩略语 (Definitions, Acronyms, and Abbreviations)

SRS: 软件需求规格说明书 (Software Requirements Specification)。

发布者 (Requester): 在平台上发布互助需求、二手商品或组队邀请的用户。

接单者 (Provider): 在平台上接受任务、购买商品或响应组队邀请的用户。

校园认证: 通过学生证、学信网或校园Edu邮箱进行的用户真实在校身份验证。

1.4 参考资料 (References)

《IEEE Std 830-1998 软件需求规格说明书推荐标准》

《中华人民共和国个人信息保护法 (PIPL)》

1.5 概述 (Overview)

本文档第2章将介绍系统的总体描述，包括产品环境、用户特征及约束条件；第3章将详细列出外部接口、功能需求、性能需求以及系统质量属性等具体需求。

2. 总体描述 (Overall Description)
2.1 产品愿景 (Product Perspective)

本平台是一个独立运作的C2C（Client to Client）校园服务社区。系统采用前后端分离架构，移动端采用跨平台技术或微信小程序（依托微信庞大的学生用户群），后端部署于云服务器。系统需集成第三方支付接口（微信/支付宝）、短信/邮件服务以及对象存储服务（OSS）。

2.2 产品功能概述 (Product Functions)

用户与认证模块： 注册登录、校园认证、个人主页、信用分管理。

需求发布与接单模块（核心四大场景）：

快递代取： 填写取件码、大小、送达地、悬赏金。

学习辅导： 设定科目、时间、线上/线下、报酬（或技能互换）。

二手交易： 上传图文、标价、新旧程度、议价/直接购买。

组队匹配： 比赛/项目名称、招募角色、截止时间。

大厅与搜索模块： 分类浏览、按标签/关键词检索、智能推荐。

订单流转模块： 状态机管理（待接单、进行中、待确认、已完成、已取消、申诉中）。

即时通讯（IM）模块： 保护隐私的前置沟通、系统通知。

评价与反馈模块： 互评机制、信誉积分增减。

2.3 用户特征 (User Classes and Characteristics)

普通学生用户： 熟悉移动互联网，对价格和效率敏感，碎片化时间多。需要系统具有较低的学习成本和极高的界面友好度，注重隐私和交易安全。

系统管理员： 平台运营人员，具备基本的计算机操作能力。负责用户认证审核、违规内容下架、交易纠纷仲裁以及数据统计。

2.4 运行环境 (Operating Environment)

客户端： 支持微信小程序环境；或主流移动操作系统 iOS 14+ 及 Android 10+。

管理后台： 支持现代主流浏览器（Chrome, Edge, Safari）。

服务端： Linux (Ubuntu/CentOS)，MySQL 8.0+，Redis，Node.js/Java/Python 运行环境。

2.5 设计和实现约束 (Design and Implementation Constraints)

安全与合规： 必须强制实行实名/校园身份认证才可发布和接单；必须遵守相关法律法规，禁止发布代课、代考等违规违纪需求。

支付约束： 平台初期不直接触碰资金池，可采用“平台虚拟担保（冻结积分/保证金）+ 线下结算”或接入合规的第三方支付担保交易接口。

2.6 假设和依赖关系 (Assumptions and Dependencies)

假设学校允许学生在非上课时间进行合规的校内互助活动。

系统依赖第三方实名认证 API、支付 API 及地图/定位 API（用于距离测算）。

3. 具体需求 (Specific Requirements)
3.1 外部接口需求 (External Interfaces)

3.1.1 用户界面 (UI): 移动端采用年轻态、扁平化的UI设计。主页必须包含清晰的“快递”、“辅导”、“二手”、“组队”四个大入口及一个悬浮的“+”发布按钮。

3.1.2 硬件接口: 调用手机摄像头（拍摄二手商品/取件码）、调用GPS模块（获取校内定位，如宿舍楼栋）。

3.1.3 软件接口:

微信/支付宝支付 SDK。

第三方即时通讯云服务（如腾讯云 IM）。

阿里云/腾讯云 OSS（用于存储用户头像及商品图片）。

3.1.4 通信接口: 客户端与服务端采用 HTTPS 协议进行 RESTful API 交互；聊天功能采用 WebSocket 协议。

3.2 功能需求 (Functional Requirements)
3.2.1 用户与认证模块 (User & Authentication)

REQ-U-01: 系统应支持手机号验证码注册与微信一键授权登录。

REQ-U-02: 系统应强制要求用户进行“校园认证”（上传学生证/校园卡或输入 edu 邮箱），未认证用户仅可浏览，不可发布、接单或留言。

REQ-U-03: 系统应为每位用户建立“信用积分”（初始100分），完成订单增加分，爽约或被举报扣除分。积分低于60分将限制接单权限。

3.2.2 需求大厅与搜索模块 (Marketplace & Search)

REQ-M-01: 首页应按时间倒序或按位置距离展示最新需求卡片。

REQ-M-02: 用户可按类别（快递、辅导等）、状态（急单、高佣金）进行筛选。

REQ-M-03: 系统支持模糊搜索，用户可通过关键词搜索需求内容。

3.2.3 需求发布与接单模块 (Task Posting & Accepting)

REQ-T-01 【快递代取】:

发布：发布者需输入快递点位置、取件码（支持打码保护）、包裹规格（小/中/大）、送达宿舍楼栋及悬赏金额。

接单：接单者点击接单后，取件码才完整显示给接单者。

REQ-T-02 【学习辅导】:

发布：支持发布“求教”或“授课”。需填写科目名称、时间安排、预期报酬及个人基础。

接单：需通过IM双方确认细节后，由一方发起“达成互助”确认。

REQ-T-03 【二手交易】:

发布：发布者需上传最多9张图片，填写物品名称、新旧程度（如9成新）、原价、转让价及交易地点。

接单：买家可直接点击“我想买”锁定商品，或使用“留言/私聊”功能进行议价。

REQ-T-04 【组队匹配】:

发布：需填写赛事/活动名称、当前队伍人数、需招募的人数及具体技能要求（如：PPT制作、数据分析）。

接单：响应者提交自己的简历/技能简介，发布者审核同意后才算组队成功。

3.2.4 订单流转管理 (Order Management)

REQ-O-01: 订单生命周期应包括：待接单 -> 进行中 -> 待确认完成 -> 已完成/已评价。

REQ-O-02: 接单者完成任务（如送达快递）后，点击“已完成”，发布者需在24小时内确认，超时系统自动确认。

REQ-O-03: 任何一方取消订单需经另一方同意；若产生纠纷，可点击“申请客服仲裁”。

3.2.5 消息与沟通模块 (Messaging)

REQ-C-01: 提供应用内文字、图片及语音聊天功能，避免双方加微信泄露隐私。

REQ-C-02: 系统消息：当需求被接单、订单状态变更、收到新留言时，系统应推送通知提醒用户。

3.2.6 后台管理模块 (Admin Dashboard)

REQ-A-01: 管理员可查看、审核、驳回用户的校园认证申请。

REQ-A-02: 管理员可监控所有发布的信息，并可对违禁词系统报警的帖子进行人工删除/封禁操作。

REQ-A-03: 客服可介入处理交易纠纷，具有修改订单状态和调整用户信用分的权限。

3.3 性能需求 (Performance Requirements)

响应时间: 大部分操作（如加载首页、发布需求）的系统响应时间应控制在 1.5 秒以内。IM 消息延迟不超过 500 毫秒。

并发量: 系统应支持单个校区至少 2000 人的同时在线活跃，峰值并发（如中午下课取快递高峰）不低于 500 QPS。

数据吞吐量: 能够支持每日新增至少 10,000 条图文数据的并发写入与读取。

3.4 逻辑数据库需求 (Logical Database Requirements)

数据库需至少包含以下实体及其关系：

Users (用户表：ID, 姓名, 学号, 手机号, 信用分, 认证状态)

Tasks (任务表：ID, 类型, 标题, 详情, 悬赏金额, 状态, 发布者ID, 接单者ID, 时间戳)

Orders (订单流转记录表)

Messages (聊天记录表)

系统必须实现逻辑删除（软删除），保留交易历史数据以备审计。

3.5 软件系统属性 (System Attributes)

3.5.1 安全性 (Security):

用户密码和敏感信息（如取件码、真实姓名、学号）必须加密存储（如使用 bcrypt 和 AES-256）。

防止 SQL 注入、XSS 攻击及跨站请求伪造 (CSRF)。

敏感词过滤：系统需内置违禁词库，自动拦截黄赌毒、代写代考等不良信息。

3.5.2 可靠性 (Reliability):

系统可用性应达到 99.9% (全年宕机时间不超过8.7小时)。

数据库需实现每日全量备份及实时增量备份。

3.5.3 可维护性 (Maintainability):

前后端代码需遵循标准的编码规范，API接口提供详细的 Swagger 接口文档。

系统设计需采用模块化思想，方便未来横向扩展（如新增“失物招领”模块）。

3.5.4 可移植性 (Portability):

后端服务需容器化（Docker），支持在主流云平台（阿里云、腾讯云、AWS）之间无缝迁移。

4. 附录 (Appendices)
4.1 待定问题 (TBD - To Be Determined)

第三方支付的具体费率及平台是否抽成的商业模式细节仍需运营团队最终确定。

跨校区、跨学校是否开放互助功能暂定为第二期需求，本期（V1.0）仅限于本校本校区内的互助。

二、 核心用例详细描述 (Detailed Use Case Descriptions)

选取了互助交易链路中最核心的 3 个用例进行详细描述。这些设计深度结合了问卷第13题（任务成功标准）及访谈笔记（历史评价、信誉体系、防诈骗）的调研结果。

1. 核心用例一：发布互助任务（UC4）

用例名称： 发布互助任务

参与者： 已认证学生（作为需求发布方）

前置条件： 用户已登录，已通过“校园实名认证”，且当前账户信用分高于限制阈值（如 ≥60分）。

基本流（正常流程）：

用户在平台首页点击“+ 发布需求”按钮。

用户选择任务类别（快递跑腿、二手交易、学习辅导、组队匹配）。

系统展示对应类别的表单。用户填写任务标题、详情、截止时间、交接地点。

用户设定悬赏（输入愿意支付的积分或接入支付输入金额）。

用户点击“确认发布”。系统将悬赏金/积分进行平台预扣托管。

系统进行自动化违禁词检测，检测通过后将需求卡片展示在需求大厅中。

替代流（异常或分支流程）：

4a. 余额/积分不足： 系统提示用户余额不足，跳转至充值或积分获取页面。

6a. 包含敏感词/违规词汇： 系统拦截发布请求，提示“包含违规内容”，要求用户修改后重新提交。

后置条件： 系统在数据库中生成一条新的订单记录，状态标记为 待接单，预扣用户的悬赏进入托管账户。

2. 核心用例二：接受互助任务（UC6）

用例名称： 接受互助任务

参与者： 已认证学生（作为任务接单方）

前置条件： 用户已登录且完成认证；信用分符合接单标准；目标任务状态必须为 待接单。

基本流（正常流程）：

用户在需求大厅浏览，点击感兴趣的任务卡片进入详情页。

用户查看具体需求内容及发布者的历史评价与信用分（响应访谈对象2、3的需求）。

用户点击“立即接单”按钮。

系统验证接单合法性，锁定该订单，状态变更为 进行中。

系统给发布者推送通知“您的任务已被接取”。

系统自动为双方创建站内私信会话（保护隐私，响应问卷第12题），并解锁隐藏的敏感信息（如取件码原图）。

替代流（异常或分支流程）：

3a. 任务已被抢单： 当用户点击接单时，系统若判定已被他人提前0.1秒抢单，则弹窗提示“手慢了，该任务已被接取”，刷新页面。

3b. 接单上限限制： 为防恶意刷单（响应问卷防范机制），系统若检测到用户当前进行中订单超过 5 单，提示“进行中任务过多，请完成后再接单”。

后置条件： 任务状态变更为 进行中，接单人与发单人身份绑定，开启私信沟通渠道。

3. 核心用例三：确认任务完成与双向互评（UC9 & UC10 合并场景）

用例名称： 确认任务完成与评价

参与者： 已认证学生（发布方与接单方）

前置条件： 接单方已执行完任务，点击了“提交完成”并上传了凭证（如包裹放在寝室门口的照片），订单状态变更为 待确认完成。

基本流（正常流程）：

系统向发布方发送“接单人已完成任务，请确认”的推送通知。

发布方查看站内信及接单方上传的完成凭证照片。

发布方核实验收无误后，点击“确认完成”。

系统将托管的悬赏（积分或金额）正式结算，打入接单方账户。

订单状态变更为 已完成，系统弹窗邀请双方进行 1~5 星打分并填写评语。

双方提交评价后，系统自动更新双方的“历史评价记录”及“信用分”。

替代流（异常或分支流程）：

2a. 任务未达标产生纠纷： 发布方核实发现任务未完成或有损坏，点击“驳回”。系统提示双方先私信沟通；若沟通无果，任一方可点击“申请客服仲裁”，订单进入 申诉中 状态，等待管理员（Admin）介入。

3a. 发布方超时未确认： 接单方提交完成后，若发布方超过 24 小时未确认且未驳回，系统将自动确认完成并结算悬赏（响应问卷第12题的拖延防止机制）。

后置条件： 订单闭环，财务/积分完成转移，双方信用档案数据得到更新。
